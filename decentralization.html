The reality

Information is scattered across different files and often on different data management systems

How we deal with it

At we distinguish between the File and its Content.
The Content nodes and File nodes are represented in a knowledge graph

So how would we represent this?

We use a hashing mechanism so the same content will always translate to the same unique id

We can use this technique to always create the same id when we wish it to be so and in contrast create a new id when this is desired

The file id is generated from a combination of what system it is stored on and what is it's path

The content id is generated from the file content

This approach is completely flexible

One file can point to several content nodes (different versions) even though its name hasn't changed.

One content node can point to several cloud locations - one for each file.

So now we can deal with multiple representations of the same file. But what about the relationships between the individual content nodes?

Document fragments

The content node is the most overall representation of a document

The any subset of a document is referred to as a Fragment and those can have sub-fragments in any arbitrary depth

Information such as the page number or the fragment's position relative to its parent can be expressed as properties on the individual fragments

A BIM Model is not any different

The same goes for a CAD document, an e-mail, an image etc.

In the BIM model already encapsulates a lot of contextural information like relationships between zones, elements etc.

But how about when an object in one model has a relationship to an object in another model?

Or when there are duplicate representations of the same object across models

Or when there is a reference between a BIM object and some other document like an image or an e-mail?

This can all be represented in the knowledge graph!
