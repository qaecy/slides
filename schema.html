The QAECY Ontology

How project knowledge is modelled in the graph

dev.qaecy.com/ont  ·  dev.qaecy.com/enum

Five layers of the model

  1. File — where content lives and how many copies exist
  2. Fragments — any subset of a document, at any depth
  3. Selectors — precise pointers into content
  4. Entity Mentions — unique occurrences of named things
  5. Canonical Entities — the single source of truth for each thing

Each layer builds on the previous.
Reasoning then stitches them together with inferred relationships.

1. File Content & File Location

Separating what a file contains from where it lives

A qcy:FileContent node represents the immutable content of a file.

Its identity is derived from the content hash — the same bytes always produce the same node.

qcy:FileContent carries metadata and category assignments

A qcy:FileLocation node represents one specific copy of the content on a storage system.

One qcy:FileContent can have many qcy:FileLocation nodes — one per DMS, cloud provider, or local path.

Same content — three locations on different systems

What this enables

  • One file present on SharePoint, Aconex and a local drive is modelled once
  • Renaming or moving a file does not break the content identity
  • qcy:FileLocation also carries qcy:mentions relationships — entity mentions inferred directly from the file path

Key data properties on qcy:FileContent

  • qcy:md5Hash — content fingerprint
  • qcy:mime — MIME type string
  • qcy:sizeBytes — file size
  • qcy:openFormat — readable without proprietary tools?
  • qcy:language — detected language
  • qcy:textSummary — LLM-generated summary

Key data properties on qcy:FileLocation

  • qcy:filePath — relative path on the provider
  • qcy:suffix — file extension
  • qcy:remoteRelativePath — provider-specific path
  • qcy:remoteProvider → qcy:ProviderType
    (Azure, AWS, GCP, Aconex, local, …)

2. Fragments & Selectors

Every subset of a document is a first-class citizen

A qcy:Fragment is any subset of a qcy:FileContent.

Fragments nest to arbitrary depth via qcy:containsFragment.

Fragment subtypes

  • qcy:DocumentPageFragment — a page in a paginated document
  • qcy:TextFragment — a passage of text
  • qcy:ImageFragment — an image or figure
  • qcy:TableFragment — a table with rows and columns

Document → Pages → sub-fragments at any depth

A qcy:Selector is a precise pointer that specifies where in a parent object a child object occurs.

Every selector has:
qcy:selectorSubject → the containing object
qcy:selectorObject → the located object
qcy:value → the positional data

Selector types

  • qcy:PageSelector — page number (integer)
  • qcy:XYWHSelector — bounding box on a page
  • qcy:TextIndexSelector — character range "start,end"
  • qcy:TextDirectiveSelector — text fragment syntax #::text=…
  • qcy:IFCExpressIDSelector — IFC line number
  • qcy:IFCGlobalIDSelector — IFC GlobalId
  • qcy:IDSelector — generic identifier (XML, JSON, …)
  • qcy:TimeRangeSelector — time range in media files

qcy:PageSelector locates the page, qcy:XYWHSelector locates the image within it

Reasoning derives qcy:containsFragment implicitly from selectors — you never need to assert it manually. 

INSERT { GRAPH <implicit> { ?sub qcy:containsFragment ?obj } }
WHERE {
  ?sel a qcy:Selector ;
       qcy:selectorSubject ?sub ;
       qcy:selectorObject  ?obj .
  ?obj a qcy:Fragment .
  FILTER NOT EXISTS { ?sub qcy:containsFragment ?obj }
}

3. Entity Mentions

Every occurrence of a named thing in a document is unique

A qcy:EntityMention is one specific occurrence of a named entity within an information object.

Two mentions of "Albert Einstein" on the same page are two distinct entity mention nodes.

Entity mentions carry:

  • qcy:label — the text as it appears in the document
  • qcy:hasEntityCategory → qcy:EntityCategory
    e.g. Person, Organization, Location, Wall, Storey, …
  • qcy:confidence — extraction confidence 0–1
  • qcy:hasProperty → qcy:Property — additional key-value facts

The containing fragment qcy:mentions the entity mention.

A qcy:Selector then specifies exactly where in that fragment the mention occurs.

Two entity mentions in a text fragment, each located by a qcy:TextIndexSelector

BIM entities are a special kind of mention

qcy:BIMEntity is a subclass of qcy:EntityMention.

Instead of a text index, selectors use qcy:IFCExpressIDSelector or qcy:IFCGlobalIDSelector.

Spatial relations extracted from IFC (qcy:zoneContainsSpace, qcy:elementHasMaterial, …) are stored directly on the entity mentions.

Reasoning derives qcy:mentions implicitly from selectors — fragments automatically inherit the entity mentions of their sub-fragments. 

INSERT { GRAPH <implicit> { ?sub qcy:mentions ?obj } }
WHERE {
  ?sel a qcy:Selector ;
       qcy:selectorSubject ?sub ;
       qcy:selectorObject  ?obj .
  ?obj a qcy:EntityMention .
  FILTER NOT EXISTS { ?sub qcy:mentions ?obj }
}

4. Canonical Entities

One node to rule all mentions of the same real-world thing

A qcy:CanonicalEntity is the standardized representation of a named entity.

An qcy:EntityMention resolves to exactly one canonical entity via qcy:resolvesTo.

Two creation modes

Bottom-up (AI)

All entity mentions are processed by the pipeline. Similar mentions are clustered and resolved to a canonical entity.

qcy:explicit = false

Top-down (User)

A project practitioner defines a list of canonical entities — e.g. from a room programme or member directory. Only mentions matching this list are resolved.

qcy:explicit = true

Multiple mentions of the same location resolve to one qcy:CanonicalEntity

Key properties on qcy:CanonicalEntity

  • qcy:label — preferred English label
  • qcy:explicit — user-confirmed entity?
  • qcy:hasFoundingMention → the mention that established this entity
  • qcy:hasEntityCategory → category (Person, Wall, Space, …)
  • qcy:similarTo → linked open data (Wikidata, OpenStreetMap, …)
  • qcy:hasProperty → additional key-value facts
  • qcy:confidence — resolution confidence

Canonical entities also inherit the relations found between their entity mentions: 

INSERT { GRAPH <implicit> { ?c1 ?rel ?c2 } }
WHERE {
  ?m1 a qcy:EntityMention ; ?rel ?m2 ; qcy:resolvesTo ?c1 .
  ?m2 a qcy:EntityMention ;              qcy:resolvesTo ?c2 .
  FILTER(?m1 != ?m2)
  FILTER NOT EXISTS { ?c1 ?rel ?c2 }
}

Relations like qcy:elementHasMaterial, qcy:contractInvolvesParty, or qcy:associatedWith automatically propagate from mentions to canonical entities.

5. Reasoning

Inferred relationships that tie the layers together

After the five layers are in place, a set of SPARQL INSERT … WHERE queries populates an implicit graph with derived facts.

The most valuable inferred relationship is qcy:about.

qcy:about connects any qcy:InformationObject directly to a qcy:CanonicalEntity.

It answers the question: "What is this document (or fragment) about?"

How qcy:about propagates

  1. Fragment mentions EntityMention
    + EntityMention resolvesTo CanonicalEntity
    → Fragment about CanonicalEntity
  2. Document containsFragment Fragment
    + Fragment about CanonicalEntity
    → Document about CanonicalEntity
  3. Transitivity extends this through any depth of nesting

Dashed edges are inferred — never asserted explicitly.
Both the page and the document become qcy:about the canonical entity.

All reasoning queries at a glance

  • rdfs:domain inference — auto-type subjects
  • rdfs:range inference — auto-type objects
  • rdfs:subClassOf inference — propagate types up the hierarchy
  • rdfs:subPropertyOf inference — propagate relations to super-properties
  • qcy:containsFragment — derived from selectors
  • qcy:mentions — derived from selectors
  • qcy:implicitlyContainsFragment — transitive containment
  • qcy:mentions — propagated through contained fragments
  • qcy:about — via direct mention → resolvesTo chain
  • qcy:about — propagated through contained fragments
  • qcy:about — via file location mentions
  • qcy:relatedEntity — lifted from mentions to canonical entities

What the graph enables

  • "List all documents about Storey 2" — query qcy:about
  • "Which elements are made of concrete?" — query qcy:elementHasMaterial on canonical entities
  • "What is this document about?" — read qcy:about on the qcy:FileContent
  • "Trace entity X back to its source" — walk qcy:hasFoundingMention → qcy:selectorObject → fragment → document
